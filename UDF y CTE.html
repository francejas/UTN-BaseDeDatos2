Funciones en MySQL

Una función en MySQL es un bloque de código SQL que recibe cero o más argumentos, realiza operaciones y devuelve un único valor. Se invoca con SELECT o dentro de expresiones. Encapsulan lógica reutilizable.

Tipos de funciones (objeto de estudio)

  • Escalares (UDFs): Devuelven un único valor. Ejemplo: calcular_descuento(precio).
  • Agregadas: Operan sobre conjuntos de filas y devuelven un resumen: SUM, AVG, MAX, MIN, COUNT.

Funciones escalares – ejemplos comunes

  • Matemáticas: ABS(x), ROUND(x,d), SQRT(x).
  • Cadenas: UPPER(str), LOWER(str), CONCAT(str1, str2).
  • Fecha: NOW(), DATE_ADD(...), YEAR(date).

Funciones agregadas

Operan sobre varias filas y devuelven una sola fila resumen: MAX(), MIN(), SUM(), AVG(), COUNT().

Sintaxis básica de una UDF

CREATE FUNCTION nombre_funcion(param1 tipo, param2 tipo, ...)
RETURNS tipo_dato_retorno
DETERMINISTIC
BEGIN
    -- lógica de la función
    RETURN valor;
END;

Ejemplo: Calcular descuento (UDF determinística)

CREATE FUNCTION calcular_descuento(precio DECIMAL(7,2))
RETURNS DECIMAL(7,2)
DETERMINISTIC
BEGIN
    DECLARE descuento DECIMAL(7,2);
    SET descuento = precio * 0.1;
    RETURN precio - descuento;
END;

Determinismo: DETERMINISTIC vs NO DETERMINISTIC

Pregunta clave: Si una función recibe exactamente los mismos valores en sus parámetros, ¿es siempre determinística?
Respuesta: Depende de lo que haga la función. Una función es determinística si, dados los mismos valores de entrada, produce siempre el mismo resultado, sin importar cuándo se ejecute ni el estado de otras tablas o variables del sistema. Por el contrario, una función es no determinística si, incluso con los mismos parámetros, puede devolver resultados diferentes (por ejemplo, si usa NOW(), RAND(), o consulta datos que pueden cambiar entre ejecuciones).

Función determinística: Ejemplo: ABS(-5) siempre devuelve 5. Una UDF que multiplica un precio por un porcentaje fijo, sin acceder a tablas ni funciones variables, también es determinística.

Función no determinística: Ejemplo: NOW() devuelve la fecha y hora actual, que cambia en cada ejecución aunque no tenga parámetros. También una función que lea el saldo de una cuenta desde una tabla podría devolver valores distintos si la tabla se modifica entre llamadas.

Importancia del determinismo:
  • Optimización: el motor puede cachear resultados de funciones determinísticas.
  • Consistencia en replicación: MySQL necesita saber si una función es predecible para mantener la coherencia maestro-esclavo.
  • Buenas prácticas: si no se especifica DETERMINISTIC o NOT DETERMINISTIC, MySQL asume NO DETERMINISTIC por defecto, lo que puede impedir ciertas optimizaciones.
Para crear una función en MySQL se debe indicar al menos una de: DETERMINISTIC, NO SQL o READS SQL DATA.

¿Cuándo usar una función?

  • Cálculos simples o validaciones.
  • Reutilización en expresiones SQL.
  • Cuando se necesita un único valor de retorno.

¿Cuándo usar un procedimiento almacenado?

  • Operaciones complejas (INSERT/UPDATE/DELETE en múltiples tablas).
  • Transacciones y control de flujo (IF, WHILE, bucles).
  • Procesos de negocio completos.

Ejemplo integrado: Función + Procedimiento

-- Función que calcula precio final (determinística si los datos de ventas no cambian durante la consulta,
-- pero en rigor, al leer de una tabla, MySQL la trataría como READS SQL DATA, no puramente determinística.
-- Se muestra solo como ejemplo conceptual.
CREATE FUNCTION calcular_precio_final(venta_id INT, descuento DECIMAL(7,2), impuesto DECIMAL(7,2))
RETURNS DECIMAL(10,2)
READS SQL DATA
BEGIN
    DECLARE subtotal DECIMAL(10,2);
    SELECT precio * cantidad INTO subtotal FROM ventas WHERE venta_id = venta_id;
    RETURN subtotal * (1 - descuento) * (1 + impuesto);
END;

-- Procedimiento que actualiza usando la función
CREATE PROCEDURE actualizar_precio_final(venta_id INT, descuento DECIMAL(7,2), impuesto DECIMAL(7,2))
BEGIN
    UPDATE ventas 
    SET precio = calcular_precio_final(venta_id, descuento, impuesto)
    WHERE venta_id = venta_id;
END;

-- Llamada
CALL actualizar_precio_final(10, 0.1, 0.21);
Restricción importante: En MySQL, las funciones definidas por el usuario no pueden realizar operaciones DML (INSERT, UPDATE, DELETE) directas sobre tablas. Están diseñadas para cálculos y lecturas de datos (con READS SQL DATA). Si se necesita modificar datos, se debe usar un procedimiento almacenado.

Otros tipos de funciones en MySQL (resumen)

  • Ventana / Analíticas: ROW_NUMBER(), RANK(), SUM() OVER() — cálculos sobre particiones sin agrupar.
  • JSON: JSON_EXTRACT(), JSON_ARRAYAGG(), JSON_OBJECT().
  • Información: USER(), DATABASE(), VERSION().
  • Control de flujo: IF(), CASE, COALESCE().
  • Seguridad y cifrado: SHA2(), MD5().
  • Espaciales (geométricas): ST_DISTANCE(), ST_CONTAINS().

Introducción a las CTE (Common Table Expressions)

Una CTE es una tabla temporal definida dentro de una consulta, utilizando la cláusula WITH. Mejora la legibilidad y permite reutilizar subconsultas, además de soportar recursividad.

Tipos de CTE

  • CTE simples: definición temporal sin autoreferencia.
  • CTE recursivas: se refieren a sí mismas, ideales para jerarquías (organigrama, árbol genealógico).

Sintaxis básica (CTE simple)

WITH mi_cte AS (
    SELECT columnas FROM tabla WHERE condicion
)
SELECT * FROM mi_cte;

Sintaxis CTE recursiva (ejemplo conceptual)

WITH RECURSIVE nombre_cte AS (
    -- ancla (punto de partida)
    SELECT col1, col2 FROM tabla WHERE condicion_inicial
    UNION ALL
    -- parte recursiva
    SELECT t.col1, t.col2 FROM tabla t
    JOIN nombre_cte n ON t.parent_id = n.id
)
SELECT * FROM nombre_cte;

Ventajas de las CTE

  • Legibilidad: dividen consultas complejas en bloques lógicos.
  • Reutilización: la misma CTE puede referenciarse múltiples veces.
  • Recursividad: esencial para datos jerárquicos.

Ejemplos prácticos de CTE

Ejemplo 1: Promedio de salarios > 3000

WITH EmpleadosSalarios AS (
    SELECT empleado_id, salario FROM Empleados WHERE salario > 3000
)
SELECT AVG(salario) AS PromedioSalario FROM EmpleadosSalarios;

Ejemplo 2: CTE con JOIN

WITH EmpleadosDepartamentos AS (
    SELECT e.empleado_id, e.nombre, d.nombre_departamento
    FROM Empleados e
    JOIN Departamentos d ON e.departamento_id = d.departamento_id
)
SELECT * FROM EmpleadosDepartamentos ORDER BY nombre_departamento;

Ejemplo 3: CTE Recursiva (jerarquía de empleados)

WITH RECURSIVE jerarquia_empleados AS (
    -- empleado raíz (sin jefe)
    SELECT id, nombre, id_jefe FROM empleados WHERE id_jefe IS NULL
    UNION ALL
    SELECT e.id, e.nombre, e.id_jefe
    FROM empleados e
    JOIN jerarquia_empleados je ON e.id_jefe = je.id
)
SELECT * FROM jerarquia_empleados;

Ejemplo comparativo: cliente con mayor facturación

Subconsulta:

SELECT cliente_id, nombre FROM clientes 
WHERE cliente_id = (SELECT cliente_id FROM ventas 
                    GROUP BY cliente_id 
                    ORDER BY SUM(total) DESC LIMIT 1);

CTE equivalente:

WITH ventas_totales AS (
    SELECT cliente_id, SUM(total) AS total_ventas
    FROM ventas GROUP BY cliente_id
)
SELECT c.cliente_id, c.nombre
FROM clientes c
JOIN ventas_totales v ON c.cliente_id = v.cliente_id
ORDER BY v.total_ventas DESC LIMIT 1;
Recomendación: Usa CTE cuando necesites legibilidad, reutilización de resultados intermedios o recursividad. Usa subconsultas para casos simples, compatibilidad con versiones antiguas o consultas muy cortas.

ElementoRelación con UDF¿Qué puede hacer?¿Qué NO puede hacer?
Funciones (UDF) Autónoma, invocable en SP, triggers, vistas, CTE Devolver un valor, usarse en SELECT/WHERE, transformaciones Operaciones DML (INSERT/UPDATE/DELETE), manejo transaccional, handlers
Procedimientos (SP) Pueden invocar UDF para cálculos específicos Ejecutar tareas complejas, DML, transacciones, cursores No se invoca desde SELECT, no devuelve valor directamente como función
Triggers Pueden usar UDF para lógica de cálculo Responder automáticamente a eventos DML No se invocan directamente, dependen de eventos de tabla
Handlers Sin relación directa (exclusivos de SP/Triggers) Manejar errores (NOT FOUND, EXIT) en cursores No aplican en UDF, vistas ni CTE
Vistas Pueden utilizar UDF dentro de su definición Ofrecer abstracción, simplificar acceso a datos No aceptan parámetros, no realizan DML directamente
CTE Una UDF puede ser invocada dentro de una CTE Crear subconsultas temporales reutilizables, jerarquías No almacenan datos de forma permanente, solo existen durante la consulta
Vistas Materializadas No directa, pero pueden basarse en lógica de UDF Almacenar resultados de consulta, mejorar rendimiento Requieren actualización manual o programada, no admiten actualización directa
Conclusión final: Las funciones determinísticas mejoran el rendimiento y la previsibilidad del código. Las CTE, junto con las UDFs bien diseñadas, permiten escribir SQL moderno, mantenible y eficiente. Siempre especifica si una función es DETERMINISTIC o NOT DETERMINISTIC para ayudar al optimizador y a otros desarrolladores.
